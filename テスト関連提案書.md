## 2. 追加リスクC（リンク先取得失敗）の事前動作確認

### 本人の要望

> 構築時の事前動作確認を入れることでリスクを抑制したい。読めなかった場合は記事のリンクだけ残してその旨を追加する。

### 対応方針

**構築時に「リンク先到達性テスト」を実施するステップを手順書に追加する。** また、タスク定義ファイルにも「取得失敗時のフォールバック動作」を明確に記述する。

---

### 追加STEP：リンク先到達性テスト（STEP 3-A として挿入）

#### 目的

本番運用開始前に、**マッキンゼー・BCG関連メールに頻出するリンク先ドメインに対して、WebFetchで実際にコンテンツを取得できるかを検証**する。

#### テスト手順

**1. テスト対象ドメインのリストアップ**

まず、過去のメールを手動で確認し、頻出するリンク先ドメインをリストアップする。

`config/test-domains.json` を作成：

```json
{
  "test_urls": [
    {
      "domain": "mckinsey.com",
      "test_url": "https://www.mckinsey.com/featured-insights",
      "expected": "記事一覧ページ。タイトルと概要が取得できるはず"
    },
    {
      "domain": "bcg.com",
      "test_url": "https://www.bcg.com/publications",
      "expected": "パブリケーション一覧。タイトルと概要が取得できるはず"
    },
    {
      "domain": "hbr.org",
      "test_url": "https://hbr.org/",
      "expected": "ペイウォールあり。冒頭のみ取得可能の可能性"
    },
    {
      "domain": "ft.com",
      "test_url": "https://www.ft.com/",
      "expected": "ペイウォールあり。取得不可の可能性が高い"
    },
    {
      "domain": "nikkei.com",
      "test_url": "https://www.nikkei.com/",
      "expected": "一部記事はペイウォール。無料記事は取得可能"
    }
  ]
}
```

**2. Claude Code でテスト実行**

Claude Code のセッション（手動）で以下のプロンプトを実行：

```
config/test-domains.json を読み込み、各 test_url に対して
WebFetch を実行してください。

各URLについて以下を記録してください：
1. 取得成功/失敗
2. 取得できた場合：ページタイトル、本文の最初の200文字
3. 取得できなかった場合：エラーの種類（タイムアウト、403、ペイウォール等）
4. 取得できたコンテンツの有用性（記事の中身が読めるか、目次だけか）

結果を config/domain-test-results.md に保存してください。
```

**3. テスト結果の評価と分類**

テスト結果に基づき、各ドメインを3つのカテゴリに分類する：

| カテゴリ | 意味 | 運用時の扱い |
|---|---|---|
| 🟢 **取得可能** | WebFetchで本文が取得できる | 通常通り要約を生成 |
| 🟡 **部分取得** | 冒頭のみ/目次のみ取得可能 | 取得できた範囲で要約 + 「全文は要ログイン」と注記 |
| 🔴 **取得不可** | ペイウォール/認証必須/ブロック | **リンクのみ記載 + 「WebFetchで取得不可」と明記** |

**4. 分類結果をタスク定義に反映**

`config/domain-access-levels.json` を作成：

```json
{
  "full_access": [
    "mckinsey.com",
    "bcg.com",
    "bcghendersoninstitute.com"
  ],
  "partial_access": [
    "hbr.org",
    "nikkei.com"
  ],
  "no_access": [
    "ft.com",
    "wsj.com",
    "bloomberg.com"
  ]
}
```

### タスク定義ファイルへの反映

`tasks/daily-email-scan.md` の「手順4：リンク先の調査」セクションを以下のように改訂：

```markdown
### 手順4：リンク先の調査

#### 4-1. リンクの抽出
- メール本文中のURLを抽出する（最大 `max_links_per_email` 件/メール）

#### 4-2. ドメインのアクセスレベル判定
- `config/domain-access-levels.json` を参照し、
  各リンクのドメインがどのカテゴリに属するか判定する

#### 4-3. カテゴリ別の処理

**🟢 full_access ドメインの場合：**
- WebFetch でページ内容を取得する
- ページタイトル、記事の要点（3行以内）を抽出する
- コンサルティング業界に関連する重要情報を記載する

**🟡 partial_access ドメインの場合：**
- WebFetch でページ内容を取得する
- 取得できた範囲で要約する
- レポートに以下を付記する：
  「⚠️ 本記事は一部のみ取得可能でした。全文は元サイトでご確認ください。」

**🔴 no_access ドメインの場合：**
- WebFetch は実行しない（トークン節約）
- レポートに以下のみ記載する：
  「🔗 [記事タイトル（取得可能な場合）](URL)」
  「📌 本リンク先はペイウォール等により自動取得できません。
   手動でご確認ください。」

**未分類ドメイン（domain-access-levels.json に未登録）の場合：**
- WebFetch を試行する
- 取得できた場合：通常通り要約
- 取得できなかった場合：🔴 と同じ扱い
- レポート末尾の「改善メモ」セクションに
  「新規ドメイン検出：[ドメイン名] → アクセスレベル要分類」と記載する
```
