# インサイトデータベース（ユニバーサル）設計書

## 1. フォーマットの目的
本フォーマット（`insight-universal.md`）は、日次レポートから抽出された「提案の武器となるインサイト」を1ファイル＝1コンポーネントとして永続的・構造的に保存するための「インサイト共有データベース」用テンプレートです。

日次レポート（`report-template.md`）がフロー情報（今日の学び）であるのに対し、本フォーマットは**ストック情報（再利用可能な知的資産）**を担います。そのため、レポート側でせっかく取得した英語原文・C-suiteタグ・プラクティス軸といった**構造化情報を欠損なく引き継ぐ**ことを第一義に設計しています。

## 2. ユニバーサル（両対応）設計の思想
将来的にどのようなナレッジ管理ツール（Notion、Obsidian、データベースシステム等）にもシームレスに移行できるよう、特定ツールの記法への**「ベンダーロックイン」を完全に排除しつつ、各ツールの強みを発揮できるハイブリッド設計**を採用しています。

### Notion（構造化データベースAPI）対応への工夫
- ファイル上部の **YAML Frontmatter** において、`[[ ]]` などのMarkdown拡張記法を一切排除し、完全な標準規格フォーマットを使用しています。
- これにより、抽出スクリプト（Python等）がYAMLをParseする際のパースエラーを完全に防ぎます。
- `tags`, `c_suite`, `industry`, `firms`, `key_english_terms` などのリスト型データは、将来的にNotionへインポート（またはAPI経由でPOST）した際、自動的に「Multi-select（複数選択）」プロパティとして美しく割り振られるように設計されています。
- `sources` はオブジェクト配列とし、Notion側ではリレーショナルDBや子ページへの変換が可能な構造にしています。

### Obsidian（セカンドブレイン・ナレッジグラフ）対応への工夫
- Obsidianの特徴である「グラフビュー（思考のネットワーク化）」を実現するには、YAML内ではなく本文内に `[[ ]]`（双方向リンク）の記述が必要です。
- 本フォーマットでは、本文の末尾に `## 🔗 ナレッジリンク（Obsidian用）` という専用セクションを設け、プラクティス軸・C-suite・タグ・関連企業を `[[キーワード]]` 形式で出力させています。
- ※この `[[ ]]` 記法をNotionにそのまま取り込んでも、単なる「カッコ付きの文字列」として処理されるだけであり、システム的・視覚的エラーには一切繋がりません。

### 英語原文の永続化（新設計思想）
- トップファームの一次情報は英語であることが多く、日本語要約だけを残すと**ニュアンス・定型表現・数値の文脈が失われる**。
- そのため、本フォーマットには専用の `## 📝 原文引用と和訳` セクションを設け、日次レポート セクション4で取得した原文引用（EN）と和訳（JA）をそのまま転記する運用とした。
- これにより、将来Notionに移行しても、Obsidianで再読しても、**英語原文が一次エビデンスとしてDB内に残り続ける**。

## 3. レコード（ファイル）の構造

| 階層 | セクション名 | 役割・記述内容 |
|---|---|---|
| **属性層** | YAML Frontmatter | 検索・ソート・グラフ化のための構造化データ。タイトル、ステータス、**プラクティス軸（practice）**、重要度、**C-suiteタグ**、関連タグ、業界、ファーム、**情報源配列（sources）**、**本インサイトで押さえる英単語（key_english_terms）** など。 |
| **示唆層** | インサイト（So What?） | 情報から読み取れる業界のトレンドや構造変化の本質。`key_english_terms` に含まれる単語は初出時に `（English）` 併記する。 |
| **事実層** | ファクト（What happened?）| インサイトの裏付けとなる一次情報を箇条書きで記載。 |
| **原文層（新設）** | 原文引用と和訳（Evidence in Original Language） | 日次レポート セクション4の原文引用（EN）と和訳（JA）をそのまま転記。一次エビデンスを英語のまま永続化する。 |
| **アクション層**| 提案活用仮説 | このインサイトを具体的にどのクライアント・どの役員アジェンダに持ち込み、どういう「提案（差別化要因、インパクト）」に化けさせるかの仮説。 |
| **グラフ層**| ナレッジリンク（Obsidian用） | ファイル同士をネットワーク接続するための双方向リンク群。プラクティス軸・C-suite・タグ・ファーム・業界を `[[ ]]` 化。 |

### プラクティス軸（practice）の列挙値
日次レポート セクション3と完全に同じ8分類を使用し、レポート⇔インサイトDB間での分類ズレを防ぐ。

- `A_Strategy_CorpFinance`
- `B_Operations_SupplyChain`
- `C_Growth_MarketingSales`
- `D_People_Organization`
- `E_Technology_Digital_AI`
- `F_Risk_Resilience_Regulation`
- `G_Sustainability_EnergyTransition`
- `H_Industry_Spotlight`

旧来の粗い `category` 分類（Market/Competitor/Technology/Regulation/Client）は新設計では廃止し、`practice` に一本化した。

## 4. 推奨する運用ロードマップ

本設計に基づく、コンサルタントチーム/個人のナレッジ蓄積の推奨フローです。

1. **Phase 1: 蓄積期（MVP運用）**
   - AIにより毎日自動で `insights/YYYY-MM-DD-xxx.md` というMarkdownファイルとしてGitHubリポジトリに蓄積していく。
   - 閲覧や検索は、GitHubの検索機能や、ローカルエディタ（VS Code等）の全局検索を活用。
2. **Phase 2: 分析期（Obsidian活用）**
   - ローカルのPCにCloneしたリポジトリをObsidian等のツールで開き、点と点が繋がるナレッジグラフ（セカンドブレイン）として活用し、新規提案の着想を得る。
   - `practice` × `c_suite` の二軸でDataviewクエリを組むと、「CFOアジェンダ × Strategy&CorpFinance」のような切り口で自分のストックを即座に引き出せる。
3. **Phase 3: 組織化期（Notion連携運用）**
   - インサイトが一定数に達し、チームやプロジェクト全体で共有・検索したくなったタイミングで、Notion等にデータベースを作成。
   - GitHub Actionsなどを経由して、既存・新規の `insight-universal.md` を自動パースし、クラウドデータベースへと統合する。
   - `sources` 配列はNotionのリレーショナルDBや子ページに変換可能。
