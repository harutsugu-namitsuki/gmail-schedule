# プロジェクト振り返り報告書：gmail-schedule WebFetchアクセス失敗

**プロジェクト終了日**: 2026-04-15  
**ステータス**: クローズ（目的不達成）

---

## 1. プロジェクト概要

### 目的

McKinsey・BCGなどのコンサルティングファームや主要ビジネスメディアがGmailに配信するニュースレターを毎朝自動処理し、リンク先記事の本文まで読み込んだ「戦略的インテリジェンスレポート」を生成する。

### 想定アーキテクチャ

```
Cloud スケジュールタスク（毎朝5:30）
  └─ Gmail MCP でニュースレターを検索
       └─ メール内URLを WebFetch で取得（記事本文まで精読）
            └─ 5層構造レポート + インサイトDBを生成
                 └─ GitHub へ commit & push
```

### 核心にあった仮定

> **「AIエージェントが WebFetch でコンサルファーム・ビジネスメディアの記事URLにアクセスできる」**

---

## 2. 失敗の経緯

### 第1回テスト（2026-04-14以前）

`config/test-domains.json` に定義した5ドメインのトップページURLに対してWebFetchテストを実施。全5ドメイン失敗。

### 第2回テスト（2026-04-14 再テスト）

「前回は一時的な問題では」という期待のもと再テスト。結果は同一。

| ドメイン | エラー | 原因 |
|---|---|---|
| mckinsey.com | HTTP 403 Forbidden | Bot/WAFによる自動遮断 |
| bcg.com | HTTP 403 Forbidden | Bot/WAFによる自動遮断 |
| hbr.org | HTTP 403 Forbidden | ペイウォール + Bot遮断 |
| ft.com | Claude Code側でフェッチ拒否 | Claude Codeのドメインブロック |
| nikkei.com | HTTP 403 Forbidden | Bot/WAFによる自動遮断 |

**成功率: 0 / 5（0%）**

### 結論

2度のテストを経て、調べたいコンサルファーム（McKinsey、BCG等）のメーリスにAIクラウドエージェントからはアクセスできないことが確認された。プロジェクトをクローズする。

---

## 3. 根本原因分析

### 直接原因

| 原因 | 対象ドメイン |
|---|---|
| **WAF / Bot検知（HTTP 403）** | mckinsey.com, bcg.com, hbr.org, nikkei.com |
| **Claude Code側のドメインブロック** | ft.com |
| **ペイウォール（仮にアクセスできても本文不可）** | hbr.org, ft.com, nikkei.com（構造的障壁） |

### 構造的原因

**主要コンサルファームとビジネスメディアは、AIエージェントからの自動アクセスを意図的に遮断している。**

これは一時的な設定ミスではなく、以下の理由による意図的な設計：

1. **コンテンツ保護**: 記事本文の無断スクレイピングを防ぐ知財・ビジネス上の動機
2. **ペイウォール維持**: 有料会員制度の崩壊を防ぐ
3. **サーバー負荷管理**: Botによる大量アクセスの排除
4. **AIへの対抗意識**: LLMによるコンテンツ学習・転用への警戒

### 設計上の誤り

プロジェクト設計時に「取れるかどうかのテストを先にやろう」という判断は正しかったが、**その結果が0%だった時点でプロジェクトの前提が崩れる**という最悪ケースを十分に想定できていなかった。

```
想定: テストして取れないドメインは no_access に分類 → 他で補完
現実: 調べたいドメインが全部 no_access → 補完する「他」が存在しない
```

## 4. 学んだこと

### 技術面

| 学び | 内容 |
|---|---|
| **AIエージェントは多くの商業サイトに弾かれる** | WAFは一般的なUser-Agentだけでなく、AIエージェントの挙動パターンそのものを検出・遮断する |
| **ft.comはClaudeレベルでブロックされている** | HTTP 403以前に、Claude Codeのツール自体がフェッチを拒否する。Anthropicが意図的にブロックしている可能性が高い |
| **トップページと個別記事URLで挙動が異なる可能性はある** | ただし、今回のプロジェクト対象（コンサルファームのメーリス）がリンクする記事は、まさに最も保護されたコンテンツである |

### プロジェクト設計面

| 学び | 内容 |
|---|---|
| **前提検証を最初のステップにすべきだった** | 「WebFetchで取れる」を前提にシステム全体を設計してから検証するのは本末転倒 |
| **「取れなかった場合のフォールバック」が「取れなかった場合のプロジェクト終了」と同義だった** | no_accessのフォールバックが「メール本文のみ」である時点で、WebFetchを使わないシステムと変わらない |
| **調べたい情報源へのアクセス可否が、プロジェクトの生死を決める** | どんなに優れたパイプラインを構築しても、情報源にリーチできなければ意味がない |

## 6. 代替アプローチの可能性（参考）

今後同様の目的を達成したい場合のアイデア（このプロジェクトでは追求しない）：

| アプローチ | 実現可能性 | コメント |
|---|---|---|
| RSSフィードを情報源にする | 🟢 高 | McKinsey・BCGはRSSを公開していることがある。WebFetchより遮断されにくい |
| メール本文の要約のみに特化する | 🟢 高 | WebFetchを一切使わず、届いたメール本文だけを処理する。情報密度は下がるが確実に動く |
| Firecrawl等の専用スクレイピングサービスを使う | 🟡 中 | 有料のスクレイピングAPIは一部ペイウォールサイトに対応しているものもある。ただしコンサルファームが許容するかは不明 |
| 公式APIやRSSが存在するソース（Substack等）に絞る | 🟢 高 | AIフレンドリーな情報発信者のコンテンツだけを対象にする |

## 7. クローズ判断の根拠

- 核心的前提（WebFetch可能）が2度のテストで否定された
- 前提が崩れた状態で残るシステムは「Gmailのメール本文を読んでまとめるだけ」であり、当初目的の付加価値（記事本文の精読・引用）がゼロになる
- これ以上の調査・改修に投資するより、別の情報源・アーキテクチャから再設計するほうが合理的
