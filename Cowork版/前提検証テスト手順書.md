# 🚨 STEP −1：前提検証テスト手順書（最優先・GO/NO-GO ゲート）

> **本書の位置づけ：前回プロジェクトを葬った唯一にして最大の失敗は、核心的前提（WebFetchで記事が取れる）を最初に検証せず、システム全体を組んでから検証し、0/5 で崩壊したことである。本書は「そのミスを二度と犯さない」ための最上流ゲートである。**
>
> **参照: [振り返り報告書.md](振り返り報告書.md) セクション 3「設計上の誤り」, セクション 4「プロジェクト設計面の学び」**

---

## 0. 本テストが成立するまで、後続のどのステップにも進んではならない

本プロジェクトの核心的前提は：

> **「Claude Cowork の Chrome ブラウザ操作（Claude in Chrome）＋ ユーザーのログイン済みセッションを使えば、前回 403 で全滅した5ドメインの記事本文を取得できる」**

この前提が**実測**で成立するまで：

- [ ] 手順書の STEP 0 以降に進んではならない
- [ ] タスク定義ファイルの本番化に進んではならない
- [ ] スケジュール登録に進んではならない
- [ ] 運用開始の宣言をしてはならない

## 1. 前回失敗との同型性チェック（最重要）

### 1-1. 前回の失敗モード（振り返り報告書より）

| # | 失敗モード | 再発防止策 |
|---|---|---|
| F1 | 前提検証を最後に回し、システム全体を組んでから検証した | 本書を**STEP −1**として最上流に置く。以降のステップは本書が PASS するまで着手禁止 |
| F2 | テスト結果0%でも「取れないドメインは除外」という逃げ道があると誤認 | 本書では「0/5 なら即プロジェクト再設計」を合格基準として明記 |
| F3 | トップページでのみテストし、個別記事URL（実運用で届くもの）では試さなかった | 本書ではトップページ + 個別記事URL + 実メール由来URLの3層で検証 |
| F4 | 一度の失敗結果で「再テスト」しただけで、条件を変えなかった | 本書では「条件を変えて複数回」を明示 |
| F5 | ツールの根本的制約（Claude Codeレベルでのドメインブロック）を知らず期待していた | 本書では「Chrome 実ブラウザ経由の User-Agent はブロックされないことの経験的確認」を最初の質問とする |

### 1-2. 今回のプロジェクトが前回と同型でない理由

| 軸 | 前回 | 今回 |
|---|---|---|
| 取得経路 | Anthropic Cloud から fetch ツールで HTTP リクエスト | ユーザー Mac 上の**実 Chrome ブラウザ**を Cowork が操作 |
| User-Agent | Claude Code の識別子 → WAF で弾かれる | 通常の Chrome → 一般ユーザーと同じ |
| Cookie / 認証 | なし | **ユーザーが事前ログイン済みのセッションを流用** |
| ペイウォール | 越えられない | ユーザーが有料購読していれば越えられる |
| Claude 側ブロック | ft.com 等が Claude Code レベルでブロックされる | Claude Desktop + Chrome拡張経由は同制約の対象外の可能性 |

**ただし上記は「理論上の違い」である。本書はこれを実測で確認する。**

## 3. 検証の3層構造（前回の失敗モード F3 への対策）

前回は**トップページ1URL**でのみテストしたため、実運用の個別記事URLでの挙動を見逃した。今回は3層でテストする。

### Layer 1：トップページ到達性（最低限の疎通確認）

目的：Chrome ブラウザ経由でサイトに到達できることを確認する。

| ドメイン | 検証URL |
|---|---|
| mckinsey.com | https://www.mckinsey.com/featured-insights |
| bcg.com | https://www.bcg.com/publications |
| hbr.org | https://hbr.org/ |
| ft.com | https://www.ft.com/ |
| nikkei.com | https://www.nikkei.com/ |

### Layer 2：個別記事URL到達性（実運用に近い条件）

目的：実際のニュースレターが誘導する先は**個別記事**である。そこに到達できるかが本番。

実施時点での各サイトの最新記事（または任意の公開記事）を1本ずつ選定し、そのURLに対して取得テストを行う。

| ドメイン | 検証URL（記入欄） |
|---|---|
| mckinsey.com | `https://www.mckinsey.com/.....` |
| bcg.com | `https://www.bcg.com/.....` |
| hbr.org | `https://hbr.org/20XX/XX/.....` |
| ft.com | `https://www.ft.com/content/.....` |
| nikkei.com | `https://www.nikkei.com/article/.....` |

### Layer 3：実メール由来URL（最終確認）

目的：実運用で届くメールに含まれるURL（リダイレクタ経由の追跡URL等）をそのまま試す。

- [ ] Gmail で McKinsey・BCG の最新ニュースレターを開く
- [ ] メール本文中のリンクを1つ選んで Claude に渡し、Chrome で取得してもらう
- [ ] ニュースレター特有のリダイレクタ（`email.mckinsey.com/c/eJx...` 等）が最終URLに正しく到達するかを確認

## 4. 実施手順

### 4-1. Claude Cowork を開き、以下のプロンプトを実行する

```
下記の検証を、Chrome ブラウザ操作（Claude in Chrome）を使って実施してください。
Computer Use（画面スクリーンショット操作）へのフォールバックは今回は許可しますが、
第一選択は必ず Chrome ブラウザ操作としてください。

【Layer 1：トップページ到達性テスト】
以下の5URLそれぞれについて：
1. Chrome で URL を開く
2. ページタイトルを取得
3. 本文の最初の300文字を取得
4. 取得ステータス（成功 / 部分取得 / 失敗）と失敗理由を記録

URLリスト：
- https://www.mckinsey.com/featured-insights
- https://www.bcg.com/publications
- https://hbr.org/
- https://www.ft.com/
- https://www.nikkei.com/

【Layer 2：個別記事URL到達性テスト】
上記5サイトそれぞれで、現時点の「最新記事」または「注目記事」を1本選び、
その個別記事URLについて Layer 1 と同じ4項目を記録。

【Layer 3：実メール由来URLテスト】
Gmail コネクター（既に Claude Desktop で接続済みの想定）で
from:(publishing@email.mckinsey.com OR bostonconsultinggroup@bcg.com) の
最新メールを1件取得し、本文中のリンクを1つ選んで Chrome で取得。
同じ4項目を記録。

【出力】
すべての結果を config/domain-test-results.md に以下の表形式で保存：

| Layer | ドメイン | URL | タイトル | 本文先頭300字 | 取得ステータス | 失敗理由 |

さらに総合判定として：
- Layer 1 成功率 / Layer 2 成功率 / Layer 3 成功率
- GO/NO-GO 判定（合格基準: Layer 2 で 3/5 以上、Layer 3 で 1/1 成功）
を末尾に記載。

重要：ログイン要求が出た場合は「既にログイン済みのはずです」と判断し、
もう一度ページをリロードして取得を試みる。それでも失敗する場合は
「要再ログイン」として記録し、次のURLに進む。
人間への確認は求めないこと。
```

### 4-2. 結果の記録先

- `config/domain-test-results.md`（Claude が自動生成）
- 本書末尾の「実施ログ」欄にも日付・PASS/FAIL をメモ

## 5. 合格基準（GO/NO-GO）

### 🟢 GO（後続ステップに進んでよい）

以下**すべて**を満たした場合：

- Layer 1（トップページ）：**3/5 以上**が成功または部分取得
- Layer 2（個別記事）：**3/5 以上**が成功または部分取得
- Layer 3（実メール由来）：**1/1 成功**（本書で指定した1件が取れる）

### 🟡 CONDITIONAL GO（条件付きで進む）

- Layer 1 は **3/5 以上**、Layer 2 が **2/5 以下**の場合
- → **Layer 2 で失敗したドメインは即 `no_access` に分類**し、それらを除外した運用設計として STEP 0 に進む
- ただしこの場合、本プロジェクトの価値は Layer 2 で成功したドメインのみに依存する。「それで十分な情報量が得られるか」を改めて判断すること

### 🔴 NO-GO（プロジェクトを一旦止めて再設計）

以下**いずれか**に該当する場合：

- Layer 1 で **2/5 以下**しか成功しない
- Layer 3 が失敗する（実運用URLで取れない時点で、システム全体の前提が崩れる）
- Chrome ブラウザ操作そのものが安定しない（タイムアウト頻発・クラッシュ等）

この場合：

1. **後続ステップ（手順書の STEP 0 以降）には絶対に進まない**
2. [振り返り報告書.md](振り返り報告書.md) セクション 6「代替アプローチ」に記載した選択肢から再設計する：
   - RSS フィードを情報源にする
   - メール本文のみでの要約に特化する
   - Firecrawl等の専用スクレイピングサービスを使う
   - AIフレンドリーな情報発信者（Substack等）に絞る
3. 本書に「NO-GO 日付 / 理由 / 次アクション」を追記し、意思決定を記録する

> **NO-GO 判定が出たら、「とりあえず動かしてみよう」「いずれ解決するかも」で進めてはならない。前回プロジェクトを失敗させた根本原因は、まさにその「とりあえず」だった。**

---

## 6. 合格したら何をするか

GO と判定された場合のみ、以下の順で後続に進む：

1. Layer 2 の結果に基づいて `config/domain-access-levels.json` を**実測値**で初期化
2. [手順書.md](手順書.md) の STEP 0 に進む
3. 本書末尾の「実施ログ」に GO 日付と合格根拠を記録

---

## 7. 定期再検証

本書は初回だけでなく、**月1回の再実行**を義務とする。理由：

- 対象サイトが Bot対策を強化すると一夜で失敗に転じる
- Chrome のログインセッションが切れると Layer 2 が一斉に失敗する
- Claude Desktop のバージョンアップで挙動が変わる可能性がある

再実行時に合格基準を割ったら、即 NO-GO として一旦運用停止し、原因調査してから再開する。

## 付録：前回失敗の引用（本書の存在理由）

> プロジェクト設計時に「取れるかどうかのテストを先にやろう」という判断は正しかったが、**その結果が0%だった時点でプロジェクトの前提が崩れる**という最悪ケースを十分に想定できていなかった。
>
> 想定: テストして取れないドメインは no_access に分類 → 他で補完
> 現実: 調べたいドメインが全部 no_access → 補完する「他」が存在しない
>
> — [振り返り報告書.md](振り返り報告書.md) §3「設計上の誤り」

本書は、上記を踏まえて「核心的前提の検証を最上流に置き、失敗したら無条件で一時停止する」ためのゲートである。**本書を省略したら、前回と同じ失敗を繰り返す。**
